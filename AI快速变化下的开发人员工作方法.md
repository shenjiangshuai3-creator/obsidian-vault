# AI快速变化下的开发人员工作方法

> [!summary] 核心观点
> 不要把自己定位为某个模型或框架的使用者，而要成为能够把业务、数据、软件工程和 AI 可靠组合起来的开发人员。

## 一、调整工作重心

面对持续变化的 AI，开发人员不需要追完所有模型和框架，而应该建立一套“模型变化，但系统仍然可维护”的工作方法。

开发工作的价值会逐渐集中在：

- 把模糊的业务需求转成可以验证的任务
- 为 AI 准备可靠的数据、上下文和工具
- 控制权限、安全、成本和错误
- 判断哪些环节交给 AI，哪些必须使用确定性程序或人工处理
- 使用评测数据判断模型是否真正有效

单纯编写重复代码的价值会降低，但系统设计、业务理解和工程判断会更加重要。

## 二、建立稳定的 AI 开发流程

### 1. 先定义业务结果

不要从“我要做一个 Agent”开始，而要先定义明确的业务结果。

例如：

> 自动检查飞书人员数据与 MySQL 数据是否一致，并生成差异报告。

同时确定验收指标：

- 准确率
- 处理时间
- 单次运行成本
- 人工介入率
- 错误是否可以追溯和恢复

### 2. 先建立评测集，再选择模型

整理 20～100 个真实案例，其中应包括：

- 正常数据
- 异常数据
- 缺失字段
- 重复数据
- 边界情况
- 容易产生错误判断的案例

模型、提示词或工作流发生变化时，都使用同一批案例重新测试。没有评测集，模型升级基本只能依靠主观感受。

### 3. 按复杂度逐级实现

优先选择最简单、最稳定的实现方式：

```text
普通程序逻辑
→ 单次模型调用
→ 模型调用受控工具
→ 固定 AI 工作流
→ 自主 Agent
→ 多 Agent 协作
```

可以用明确代码解决的问题，不要交给不确定的模型。只有任务确实需要理解、判断或动态规划时，才增加 Agent 的自主性。

### 4. 隔离模型依赖

不要让不同厂商的模型调用散落在业务代码中，建议建立统一的 AI 服务层：

```text
业务层
  ↓
AI 服务层
  ↓
OpenAI / Gemini / 本地模型
```

以下内容应该配置化：

- 模型名称
- 提示词版本
- 超时时间
- 重试策略
- Token 和成本限制
- 输出格式

这样模型发生变化时，不需要重写整个业务系统。

### 5. 让 AI 使用受控工具

相比让 AI 直接操作数据库，更可靠的方式是提供职责明确、权限有限的工具：

```text
query_employee
compare_department
generate_report
create_approval_request
```

每个工具都需要具备：

- 参数校验
- 权限检查
- 超时与重试
- 操作日志
- 错误处理

写入、删除、发送和权限修改等高风险操作，应当经过人工确认。

## 三、需要持续建设的能力

### 基础工程能力

- Python 或 TypeScript
- SQL、HTTP API 和异步任务
- Docker、Linux 和云服务
- 数据库、缓存和消息队列
- 测试、监控、安全和权限控制

### AI 工程能力

- 上下文、Token 和结构化输出
- Tool Calling、MCP 和 Agent 工作流
- Embedding、RAG 与数据检索
- 模型评测和回归测试
- 提示注入、数据泄露和权限越界防护
- 延迟、并发和 Token 成本优化

不必把主要精力放在提示词技巧上。可靠的数据、工具设计和评测系统通常更有长期价值。

## 四、把 AI 用进日常开发

AI 适合辅助完成：

- 阅读陌生代码并生成模块说明
- 编写测试、脚本和重复性代码
- 分析日志、SQL 和异常堆栈
- 生成接口文档和数据映射
- 提供代码审查的初步意见
- 快速制作技术验证版本

开发人员仍然需要负责：

- 确认真实需求
- 审查关键代码
- 验证边界条件
- 检查数据与权限风险
- 对上线结果负责

AI 生成的代码应当按照“未经审查的外部代码”处理。

## 五、建立固定的学习和工作节奏

建议使用 `70/20/10` 分配方式：

- `70%`：解决当前真实业务问题
- `20%`：建设评测、数据和 AI 工程基础设施
- `10%`：研究新模型、新框架和论文

每周选择一个新能力进行小规模验证；每月使用固定评测集比较一次模型。不要因为新模型发布就立即迁移，只有质量、成本或速度出现明确收益时才更换。

## 六、推荐实践项目

结合飞书、AnyCross 和 MySQL，可以建设一个小而完整的人员数据质量检查服务：

> 定时同步飞书数据，自动检查缺失字段、重复人员和部门异常，生成差异报告，由人事确认后再执行修正。

这个项目可以同时练习：

- 数据同步
- MySQL 数据处理
- 模型调用
- 工具设计
- 权限审批
- 模型评测
- 日志与监控

相比单纯制作聊天机器人，这类项目更接近真实的 AI 工程工作。

## 七、行动清单

- [ ] 选择一个真实、边界清晰的业务问题
- [ ] 整理第一批真实评测案例
- [ ] 使用普通程序和单次模型调用完成最小版本
- [ ] 将模型调用封装到统一 AI 服务层
- [ ] 为外部工具增加参数校验、权限和日志
- [ ] 建立质量、延迟和成本监控
- [ ] 每月使用固定评测集评估是否需要更换模型
