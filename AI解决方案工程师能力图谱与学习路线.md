# AI解决方案工程师能力图谱与学习路线

> [!summary] 学习目标
> 具备将业务需求、数据、软件工程和 AI 组合成可靠应用的能力，而不只是会调用某个模型或使用某个框架。

## 一、能力全景

| 能力层 | 核心内容 | 最终要解决的问题 |
|---|---|---|
| 编程与接口 | Python、TypeScript、HTTP API、异步任务 | 能否构建一个可运行的服务 |
| 数据与存储 | SQL、数据库设计、缓存、消息队列 | 数据能否正确、稳定地流转 |
| 部署与运维 | Linux、Docker、云服务、监控 | 应用能否稳定上线和排障 |
| AI 基础 | 上下文、Token、结构化输出 | 模型输出能否被程序可靠使用 |
| AI 应用 | Tool Calling、MCP、Agent、RAG | AI 能否连接数据并执行任务 |
| AI 质量 | 评测集、回归测试、可观测性 | AI 效果能否被客观验证 |
| AI 安全 | 提示注入、权限、隐私和数据隔离 | AI 是否会越权或泄露数据 |
| AI 性能 | 延迟、并发、缓存和 Token 成本 | AI 应用能否经济地规模化运行 |

## 二、基础工程能力

### 1. Python 与 TypeScript

#### 学习重点

Python 作为主要学习语言，重点掌握：

- 数据类型、函数、类和模块
- 类型注解与 Pydantic
- 文件、JSON、CSV 和异常处理
- `requests`、`httpx` 与 FastAPI
- `async` / `await` 异步编程
- pytest 单元测试
- 虚拟环境和依赖管理

TypeScript 作为前端和 Node.js 补充能力，重点掌握：

- 类型、接口、泛型
- Promise 和异步调用
- Node.js API 服务
- React 基础

#### 掌握标准

- 能独立编写一个 FastAPI 服务
- 能调用第三方 API 并处理超时、重试和异常
- 能使用类型定义约束输入输出
- 能为关键业务逻辑编写测试

#### 实践任务

开发一个人员信息查询 API：输入员工 ID，查询 MySQL 并返回结构化 JSON。

### 2. SQL 与数据库设计

#### 学习重点

- 查询、聚合、窗口函数和多表 JOIN
- 主键、外键、唯一约束和索引
- 事务、隔离级别和并发更新
- 表结构与字段类型设计
- 执行计划和慢 SQL 优化
- 数据迁移、备份和恢复

#### 掌握标准

- 能根据业务实体设计规范的表结构
- 能判断查询是否使用了合理索引
- 能保证同步任务重复执行时不产生重复数据
- 能通过事务保护多表写入的一致性

#### 实践任务

设计飞书人员、部门和人员部门关系表，并实现增量同步与幂等写入。

### 3. HTTP API 与异步任务

#### 学习重点

- REST API、状态码和认证方式
- API 分页、限流、超时和重试
- Webhook 与事件驱动
- 定时任务和后台任务
- 幂等键、失败重试和死信处理
- Celery、Redis Queue 或同类任务系统

#### 掌握标准

- 能设计清晰的 API 输入输出契约
- 能处理分页拉取和接口限流
- 能避免重试导致重复写入
- 能记录每次任务的状态和失败原因

### 4. Docker、Linux 与云服务

#### 学习重点

- Linux 文件、进程、网络和权限
- 环境变量与日志查看
- Dockerfile、镜像、容器和数据卷
- Docker Compose 编排应用与数据库
- Nginx、HTTPS 和反向代理
- 云服务器、对象存储和托管数据库基础

#### 掌握标准

- 能把 API、MySQL 和 Redis 组成一个可启动环境
- 能使用日志和系统命令定位服务异常
- 能区分开发、测试和生产配置
- 不把密码和密钥写进代码或镜像

### 5. 缓存与消息队列

#### 学习重点

- Redis 常见数据结构与过期策略
- Cache Aside 缓存模式
- 缓存穿透、击穿和雪崩
- 消息生产、消费、确认与重试
- 消息重复、乱序和积压处理

#### 掌握标准

- 能判断什么数据适合缓存
- 能通过幂等设计处理重复消息
- 能使用队列解耦耗时任务
- 能监控队列积压和任务失败

### 6. 测试、监控、安全与权限

#### 学习重点

- 单元测试、集成测试和端到端测试
- 日志、指标、Trace 和告警
- 身份认证与基于角色的权限控制（RBAC）
- 密钥管理、输入校验和 SQL 注入防护
- 最小权限原则和操作审计

#### 掌握标准

- 关键业务流程具有自动化测试
- 每个请求和异步任务都可以追踪
- 能回答“谁在什么时间访问或修改了什么数据”
- 高风险操作具有审批、审计或回滚能力

## 三、AI 工程能力

### 1. 上下文、Token 与结构化输出

#### 学习重点

- System、User、Assistant 和 Tool 消息
- 上下文窗口与 Token 计算
- Prompt 模板与版本管理
- JSON Schema、Pydantic 和结构化输出
- 模型输出校验、重试与降级

#### 掌握标准

- 不依赖自然语言字符串解析关键结果
- 能控制上下文长度和无关信息
- 能验证模型返回字段的类型和取值范围
- 结构化输出失败时有明确的处理策略

#### 实践任务

把员工信息文本抽取为固定 JSON，包括姓名、部门、岗位和入职日期，并校验必填字段。

### 2. Tool Calling、MCP 与 Agent 工作流

#### 学习重点

- Tool 的名称、描述和输入 Schema 设计
- 模型选择工具和生成调用参数的过程
- MCP 的 Server、Client、Tool、Resource 和 Prompt
- Agent 的状态、规划、执行和反馈循环
- 固定工作流与自主 Agent 的边界
- Human-in-the-loop 人工审批

#### 掌握标准

- 能把数据库或 API 封装成最小权限工具
- 工具调用具有超时、参数校验和审计日志
- 能限制 Agent 的循环次数和执行范围
- 写入、删除和发送类操作必须经过授权或确认

#### 实践任务

为人员数据助手提供三个工具：查询人员、比较部门、生成差异报告。第一阶段只允许读取，不开放直接修改数据库。

### 3. Embedding、RAG 与数据检索

#### 学习重点

- Embedding 和向量相似度
- 文档加载、清洗、切分和元数据
- 向量检索、关键词检索与混合检索
- Top-K、相似度阈值和 Reranker
- 引用来源、权限过滤和数据更新
- RAG 的召回率、准确率与幻觉问题

#### 掌握标准

- 能针对 Markdown、PDF、表格分别设计切分策略
- 回答能够附带真实来源
- 不同部门只能检索其有权限访问的文档
- 能区分“没有检索到”与“模型不知道”

#### 实践任务

构建 Obsidian 笔记问答助手，回答时返回笔记名称和引用片段。

### 4. 模型评测与回归测试

#### 学习重点

- 建立真实案例评测集
- 正确性、完整性、幻觉率和工具成功率
- 确定性规则、人工评分和 LLM-as-a-Judge
- Prompt、模型和 RAG 参数的对比实验
- 线上反馈采集与失败案例沉淀

#### 掌握标准

- 模型或 Prompt 变更前后都有量化对比
- 关键案例可以自动回归
- 评测标准不只包含“回答看起来不错”
- 能根据失败类型定位是检索、模型、工具还是业务规则的问题

#### 推荐评测指标

| 场景 | 主要指标 |
|---|---|
| 信息抽取 | 字段准确率、漏提率、格式通过率 |
| RAG 问答 | 召回率、答案正确率、引用准确率、幻觉率 |
| Tool Calling | 工具选择正确率、参数正确率、执行成功率 |
| Agent 工作流 | 任务完成率、人工介入率、平均步骤数 |
| 生产服务 | P95 延迟、错误率、单次任务成本 |

### 5. 提示注入、数据泄露与权限越界防护

#### 学习重点

- 直接与间接提示注入
- 不可信网页、文档和工具返回值
- 用户权限与数据权限隔离
- 敏感信息识别和脱敏
- 工具白名单、参数约束和最小权限
- 审批、沙箱、审计和异常终止

#### 掌握标准

- 不把外部文档中的指令当成系统指令执行
- 模型不能通过自然语言绕过后端权限检查
- 敏感字段进入模型前经过必要处理
- 高风险操作有明确的人工确认步骤

> [!warning] 关键原则
> Prompt 只能表达规则，不能替代后端权限控制。真正的权限必须在工具和服务端执行。

### 6. 延迟、并发与 Token 成本优化

#### 学习重点

- 首 Token 延迟与总响应时间
- 并发控制、限流和连接池
- 流式输出、异步调用和并行工具调用
- Prompt 压缩、上下文裁剪和历史摘要
- 语义缓存与普通缓存
- 大小模型路由、批处理和失败降级

#### 掌握标准

- 可以统计每个请求的 Token、耗时和费用
- 能找出链路中最慢、最贵的环节
- 简单任务优先使用成本更低的模型或普通程序
- 超时或模型不可用时有降级方案

## 四、推荐学习顺序

### 第 1 阶段：建立工程底座

- Python、SQL、HTTP API
- FastAPI、Pydantic、pytest
- MySQL、Git、Linux、Docker

**阶段产出：** 人员信息查询和同步服务。

### 第 2 阶段：掌握可靠的模型调用

- 上下文与 Token
- Prompt 模板
- 结构化输出
- 超时、重试、日志和成本记录

**阶段产出：** 员工数据结构化抽取服务。

### 第 3 阶段：连接企业数据与工具

- Embedding 与 RAG
- Tool Calling 与 MCP
- Agent 工作流和人工审批

**阶段产出：** Obsidian 知识库问答助手和只读人员数据助手。

### 第 4 阶段：生产化

- 模型评测与回归测试
- 权限、安全与审计
- 监控、延迟、并发和成本优化

**阶段产出：** 可评测、可监控、可审计的人员数据质量 Agent。

## 五、综合项目路线

### 项目 1：飞书人员数据同步服务

```text
飞书 API / AnyCross
        ↓
字段校验与数据清洗
        ↓
MySQL 幂等写入
        ↓
同步日志与异常告警
```

主要验证：Python、API、SQL、异步任务、Docker 和监控。

### 项目 2：Obsidian 知识库问答助手

```text
Obsidian Markdown
        ↓
文档切分与 Embedding
        ↓
混合检索与 Rerank
        ↓
带引用的回答
```

主要验证：RAG、权限过滤、引用准确性和评测。

### 项目 3：人员数据质量 Agent

```text
定时触发
  ↓
读取飞书与 MySQL 数据
  ↓
规则检查 + AI 异常解释
  ↓
生成差异报告
  ↓
人事确认
  ↓
执行受控修正
```

主要验证：Tool Calling、Agent 工作流、人工审批、安全、审计和成本优化。
