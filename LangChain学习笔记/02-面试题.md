# LangChain 面试题

## 一、基础概念

### 1. LangChain 是什么？解决了什么问题？

LangChain 是 LLM 应用开发框架，用来把模型、Prompt、工具、检索、状态管理、评估监控等能力组合成完整应用。它解决的不是模型训练问题，而是“如何把模型接入真实业务流程”的问题。

### 2. LangChain 和直接调用 OpenAI / 通义 / Claude API 有什么区别？

直接调用模型 API 只能完成一次模型请求。LangChain 提供更高层的应用抽象，比如 Prompt Template、Tool、Retriever、Agent、Chain、Graph、Callback、Tracing。它适合多模型、多步骤、多工具、多数据源的应用。

### 3. LangChain 的核心模块有哪些？

常见核心模块：

- Chat Model
- Prompt Template
- Output Parser
- Tool
- Retriever
- Vector Store
- Agent
- LangGraph
- LangSmith

### 4. Prompt Template 有什么价值？

Prompt Template 可以把提示词参数化、结构化、复用化，避免提示词散落在代码里。它也方便做版本管理、AB 测试和评估。

### 5. Output Parser 为什么重要？

因为模型天然输出自然语言，业务系统通常需要 JSON、列表、分类标签、SQL 等结构化结果。Output Parser 可以降低后续解析成本，也能让失败重试更可控。

## 二、RAG 高频题

### 6. 什么是 RAG？

RAG 是 Retrieval-Augmented Generation，检索增强生成。它先从外部知识库检索相关内容，再把检索结果作为上下文交给模型生成答案，从而降低幻觉并接入私有知识。

### 7. RAG 的完整流程是什么？

```text
文档加载 -> 文本切分 -> Embedding -> 向量库存储
用户提问 -> 检索相关 chunk -> 拼接上下文 -> 模型生成 -> 返回答案和引用
```

### 8. RAG 中 chunk size 怎么设置？

没有固定答案，要看文档类型和问题粒度。chunk 太大会引入噪声，太小会丢上下文。通常需要结合 overlap、标题层级、段落结构和评估结果调参。

### 9. 为什么要设置 chunk overlap？

因为切分边界可能截断语义。overlap 可以保留相邻 chunk 的上下文，减少信息断裂，但 overlap 太大会增加存储量、召回重复和成本。

### 10. 向量检索不准怎么办？

可以从这些方向优化：

- 换更适合领域和语言的 embedding 模型
- 调整 chunk size 和 overlap
- 增加 metadata filter
- 使用关键词检索 + 向量检索的混合召回
- 加 reranker
- 清洗原始文档
- 建立评估集，按问题类型分析失败样本

### 11. RAG 如何减少幻觉？

- 要求模型只基于检索上下文回答
- 答案附带引用来源
- 检索不到时明确回答不知道
- 提高召回质量
- 对高风险问题加人工审核
- 用评估集监控准确率和幻觉率

### 12. RAG 和 fine-tuning 有什么区别？

RAG 更适合接入频繁变化的外部知识，更新成本低，可追溯来源。fine-tuning 更适合学习风格、格式、任务模式或稳定知识，但不适合频繁更新事实。企业知识库通常优先 RAG。

### 13. Retriever 和 Vector Store 是什么关系？

Vector Store 负责存储和相似度搜索 embedding 后的文档；Retriever 是面向应用的检索接口，可以封装向量搜索、关键词搜索、混合检索、过滤条件和重排逻辑。

## 三、Agent 高频题

### 14. 什么是 Agent？

Agent 是让模型根据目标和上下文自主决定下一步动作的机制。它可以选择是否调用工具、调用哪个工具、如何使用工具结果，并最终完成任务。

### 15. Agent 和 Chain 有什么区别？

Chain 通常是固定流程，步骤由开发者提前定义。Agent 的流程更动态，由模型根据中间结果决定下一步。Chain 更可控，Agent 更灵活但风险更高。

### 16. Tool 设计要注意什么？

- 名称清晰
- 描述准确
- 参数 schema 明确
- 返回值稳定
- 权限最小化
- 错误信息可理解
- 高风险动作需要确认

### 17. Agent 为什么容易失控？

因为模型可能误选工具、重复调用工具、陷入循环、误解工具返回值，或者执行了不该执行的动作。生产中要限制最大步数、加权限控制、加日志、加人工确认。

### 18. 如何提升 Agent 的稳定性？

- 工具粒度适中
- 工具描述具体
- 限制最大调用次数
- 对工具返回做结构化
- 对失败场景设置 fallback
- 使用 LangGraph 显式控制状态和分支
- 用 LangSmith 观察每一步调用

### 19. 什么场景不适合用 Agent？

- 步骤固定的审批流
- 强确定性的交易流程
- 高风险删除、支付、权限修改
- 简单问答
- 可以用普通代码明确实现的逻辑

## 四、LangGraph 高频题

### 20. LangGraph 是什么？

LangGraph 是用于构建有状态、多步骤、可控 Agent 工作流的框架。它把流程表示成图：节点执行动作，边控制流转，状态贯穿全流程。

### 21. LangGraph 和 LangChain Agent 的关系是什么？

可以理解为：LangChain Agent 适合快速创建工具调用应用；LangGraph 更适合复杂、可控、可恢复的生产级 Agent 流程。很多高级 Agent 能力底层会依赖图式编排思想。

### 22. 什么时候应该使用 LangGraph？

- 需要状态管理
- 需要分支判断
- 需要循环
- 需要人工审核
- 需要多 Agent 协作
- 需要断点恢复或 checkpoint

### 23. LangGraph 中 State、Node、Edge 分别是什么？

- State：流程共享状态，比如用户问题、检索结果、中间答案。
- Node：执行单元，比如检索、生成、审核、调用工具。
- Edge：节点之间的流转关系。

## 五、工程化与生产问题

### 24. LangSmith 有什么用？

LangSmith 用于 trace、debug、评估和监控。它可以看到每次模型调用、Prompt、检索结果、工具调用、耗时和错误，适合排查 RAG 和 Agent 的线上问题。

### 25. 如何评估一个 RAG 系统？

可以评估：

- 检索召回率
- 检索相关性
- 答案准确率
- 引用是否正确
- 幻觉率
- 延迟
- token 成本
- 用户满意度

### 26. 如何控制 LangChain 应用成本？

- 缓存重复请求
- 控制上下文长度
- 减少无效检索 chunk
- 使用合适的小模型处理简单任务
- 对复杂任务才调用强模型
- 监控 token 使用
- 对 Agent 限制最大步数

### 27. 如何处理流式输出？

流式输出适合聊天、长文本生成和需要改善用户体验的场景。后端要处理增量 token、异常中断、前端展示和最终结果落库。

### 28. 如何处理模型返回不稳定？

- 使用结构化输出
- 降低 temperature
- 加校验和重试
- 明确输出 schema
- 对关键字段做业务校验
- 建立评估集回归测试

### 29. LangChain 项目如何做权限控制？

权限控制不能只靠 Prompt。应该在工具层、检索层、数据库层做权限校验，例如按用户过滤文档 metadata、限制工具可访问资源、高风险动作二次确认。

### 30. 如何排查 RAG 回答错误？

按链路排查：

1. 用户问题是否被正确理解
2. 检索 query 是否合理
3. 召回 chunk 是否相关
4. 上下文是否过长或有噪声
5. Prompt 是否要求基于材料回答
6. 模型是否忽略上下文
7. 是否需要 rerank 或 metadata filter

## 六、项目追问题

### 31. 你做过的 LangChain 项目架构是什么？

回答模板：

```text
前端输入问题 -> 后端接收请求 -> LangChain 构造 Prompt
-> Retriever 从向量库召回文档 -> 模型基于上下文回答
-> 返回答案、引用来源、trace id -> LangSmith 记录链路
```

### 32. 你的项目遇到过什么问题？

可以讲这些：

- PDF 切分效果差
- 中文 embedding 召回不准
- 用户问题太口语化
- 检索到相似但不正确的文档
- 模型编造不存在的字段
- Agent 反复调用同一个工具
- token 成本过高

### 33. 你如何优化过？

可回答：

- 文档清洗和分层切分
- 给 chunk 增加标题、来源、业务标签
- 使用 metadata filter
- 增加 reranker
- 让答案强制引用来源
- 建立标准问答集做回归评估
- 使用 LangSmith 看 trace 定位错误环节

## 七、加分回答

### 34. 为什么说 Prompt 不能替代权限控制？

Prompt 只是模型行为约束，不是安全边界。用户可以绕过提示词，模型也可能误判。真正的权限必须在系统层实现，例如 API 鉴权、数据库权限、检索过滤、工具白名单。

### 35. 如何设计一个企业知识库问答系统？

核心设计：

- 文档接入：支持 PDF、Markdown、网页、数据库
- 文档处理：清洗、切分、加 metadata
- 索引：embedding + 向量库
- 检索：混合检索、filter、rerank
- 生成：基于上下文回答并附引用
- 权限：按用户和部门过滤文档
- 评估：标准问题集、人工标注、线上反馈
- 观测：trace、耗时、成本、错误率

### 36. Agent 项目上线前要做哪些防护？

- 工具白名单
- 最大执行步数
- 超时控制
- 参数校验
- 敏感操作二次确认
- 日志和 trace
- 错误 fallback
- 权限校验
- 成本预算

## 八、面试最后反问

可以反问面试官：

- 团队目前的 AI 应用主要是 RAG、Agent，还是内部效率工具？
- 是否已经有向量库、评估集和 LangSmith 类似的观测体系？
- 业务中更关注准确率、召回率、响应速度，还是成本？
- 现在最大的痛点是知识接入、模型效果、工程稳定性，还是权限安全？
