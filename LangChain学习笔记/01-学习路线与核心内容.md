# LangChain 学习路线与核心内容

## 一、LangChain 是什么

LangChain 是一个用于开发 LLM 应用的框架。它的价值不是“替代大模型”，而是帮助开发者把模型调用、提示词、外部工具、检索系统、状态管理、评估监控这些环节组织成可维护的应用。

典型场景：

- 企业知识库问答
- 文档总结与信息抽取
- 数据分析助手
- 自动化 Agent
- 多步骤业务流程编排
- 连接数据库、API、搜索、文件系统等外部工具

一句话理解：

> LangChain 负责把“模型能力”变成“应用流程”。

## 二、核心组件速查

| 组件              | 作用                               | 面试表达                    |
| --------------- | -------------------------------- | ----------------------- |
| Chat Model      | 调用大模型生成回复                        | LangChain 对不同模型供应商做统一封装 |
| Prompt Template | 管理提示词变量和结构                       | 把提示词模板化，便于复用、测试和版本管理    |
| Message         | 表示 system/user/assistant/tool 消息 | Chat 模型本质处理的是消息序列       |
| Output Parser   | 把模型输出转成结构化结果                     | 降低后续业务逻辑处理成本            |
| Tool            | 让模型调用外部能力                        | Agent 能做事的关键接口          |
| Retriever       | 根据 query 找相关文档                   | RAG 的检索入口               |
| Vector Store    | 存储 embedding 后的文本块               | 用相似度搜索召回上下文             |
| Agent           | 让模型自主选择下一步动作                     | 适合开放式、多工具、多步骤任务         |
| LangGraph       | 用图管理状态、分支、循环                     | 适合生产级复杂 Agent           |
| LangSmith       | Trace、调试、评估和监控                   | 工程化排障和评估工具              |

## 三、基础调用链

最简单的 LLM 应用通常包含：

```text
输入 -> Prompt -> Chat Model -> Output Parser -> 业务结果
```

关键点：

- Prompt 不只是文字，而是可以参数化、版本化的模板。
- Chat Model 返回的内容最好进入 Output Parser，避免业务代码直接依赖自然语言。
- 生产中要关注超时、重试、限流、日志、成本和隐私。

## 四、Prompt Template

Prompt Template 的作用是把提示词从散落在代码里的字符串变成可维护的模板。

建议写法：

```python
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个严谨的数据分析助手。"),
    ("user", "请解释这个指标：{metric_name}")
])

messages = prompt.invoke({"metric_name": "完播率"})
```

学习重点：

- system message 用来设定角色、边界和输出要求。
- user message 放具体任务。
- 不要把不可控的大段用户输入直接拼接到系统提示词里。
- 对结构化输出要配合 schema 或 parser。

## 五、工具调用与 Agent

Tool 是 Agent 能执行动作的接口。一个好工具应该具备：

- 明确的名字
- 清晰的描述
- 稳定的输入 schema
- 可预测的返回结果
- 明确的权限边界

Agent 的典型流程：

```text
用户问题 -> 模型判断是否需要工具 -> 调用工具 -> 读取工具结果 -> 继续推理或输出答案
```

适合 Agent 的任务：

- 需要查询外部 API
- 需要多步骤决策
- 需要根据中间结果动态选择下一步
- 工具种类多，流程不完全固定

不适合 Agent 的任务：

- 步骤固定且强约束的流程
- 对确定性要求极高的审批、支付、删除操作
- 可以用普通函数或 DAG 明确实现的逻辑

## 六、RAG 核心流程

RAG 是 LangChain 最常见的落地场景之一。

```text
原始文档
  -> 文档加载
  -> 文本切分
  -> Embedding
  -> 向量库存储
  -> 用户提问
  -> 检索相关 chunk
  -> 拼接上下文
  -> 模型生成答案
```

每一步的关键问题：

| 环节 | 关键问题 |
|---|---|
| 文档加载 | 数据源是否可靠，格式是否统一 |
| 文本切分 | chunk 太大召回不准，太小上下文不足 |
| Embedding | 模型是否适合中文和业务语义 |
| 向量库 | 是否支持过滤、更新、权限隔离 |
| 检索 | top_k、相似度阈值、metadata filter 如何设置 |
| 生成 | 是否要求引用来源，如何处理不知道 |
| 评估 | 看召回率、答案准确率、幻觉率、延迟和成本 |

RAG 常见优化：

- 调整 chunk size 和 overlap
- 使用 metadata filter 缩小检索范围
- 加 reranker 提升相关性
- 多路召回：关键词检索 + 向量检索
- 对表格、代码、PDF 分别设计切分策略
- 让模型基于引用回答，不允许编造

## 七、LangGraph 什么时候用

当普通 Agent 不够可控时，用 LangGraph。

典型需求：

- 有明确状态：任务进度、用户上下文、中间结果
- 有分支：不同意图走不同节点
- 有循环：反复检索、反思、修正
- 有人工介入：审批、确认、补充信息
- 有多 Agent 协作：规划、执行、审核分工

可以这样理解：

> LangChain 适合快速搭应用，LangGraph 适合把复杂 Agent 做成可控流程。

## 八、LangSmith 工程化

LangSmith 主要解决：

- 每次模型调用的 trace
- Prompt、输入、输出、工具调用链路追踪
- 评估集管理
- 回归测试
- 线上错误排查
- 成本和延迟分析

面试中可以这样说：

> 只做 demo 时看最终回答就够了；做生产应用时必须看每一步调用、检索结果、工具参数和模型输出，否则很难定位问题。

## 九、推荐学习计划

### 第 1 阶段：基础入门

- LangChain 是什么
- Chat Model 调用
- Prompt Template
- Message 类型
- Output Parser
- 最小可运行 demo

产出：写一个“指标解释助手”。

### 第 2 阶段：RAG

- 文档加载
- 文本切分
- Embedding
- 向量库
- Retriever
- 引用来源
- RAG 评估

产出：写一个“Obsidian 笔记问答助手”。

### 第 3 阶段：Agent

- Tool 定义
- create_agent
- 工具调用过程
- 错误处理
- 权限边界
- 防止无限循环

产出：写一个“数据表字段解释 + SQL 生成助手”。

### 第 4 阶段：LangGraph

- State
- Node
- Edge
- Conditional Edge
- Checkpoint
- Human-in-the-loop

产出：写一个“需求理解 -> SQL 生成 -> 审核 -> 执行建议”的多步骤流程。

### 第 5 阶段：面试与项目包装

- 准备 RAG 项目讲解
- 准备 Agent 项目讲解
- 准备工程化问题
- 准备失败案例和优化经验

## 十、项目案例包装模板

面试讲项目时，用这个结构：

```text
背景：为什么要做这个 AI 应用？
目标：要解决什么业务问题？
方案：LangChain 负责哪些部分？
流程：输入 -> 检索/工具 -> 模型 -> 输出
难点：召回不准、幻觉、权限、成本、延迟
优化：chunk、rerank、metadata、缓存、评估集
结果：效率提升、准确率提升、人工成本下降
```
