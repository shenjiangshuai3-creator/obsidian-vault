## 项目一：本地生活全域数仓资产治理与主题域建模项目

### 简历版

**项目名称：本地生活全域数仓资产治理与主题域建模**

**项目背景：**  
公司本地生活、投流、直播、达人、内容创意等业务线沉淀了大量 MaxCompute 表，但存在 ODS 表占比高、业务线归属不清、表注释缺失、主题域划分不统一、核心交易/线索/直播/投放链路难以复用等问题。为支撑后续 BI 报表、经营分析和数仓分层建设，对全量数仓资产进行系统盘点、分类和治理规划。

**技术栈：** MaxCompute、Python、CSV/Markdown、数仓分层、主题域建模、数据治理、数据资产盘点

**个人职责：**

- 基于本地数仓表清单和业务线映射文件，使用 Python 自动化生成 `table_asset_inventory.csv` 和资产盘点摘要文档。
- 对 **691 张数仓表** 进行分层识别，识别出 `ODS`、`DWD`、`DWS`、`ADS`、`TMP` 等分层资产，其中 ODS 表 **677 张**。
- 按业务线完成资产归类，覆盖职能后台、内容创意、投流/投放、达人、直播、销售/商业化、本地生活等 **10 类业务线**。
- 按主题域完成二级分类，沉淀视频/内容/素材、线索/私信/客户、组织/用户/权限、直播、广告/投放、订单交易、门店/经销商/POI 等 **13 个主题域**。
- 识别 **19 张未映射业务线表**、**266 张缺少中文说明表**、**63 张未分类表**，输出后续治理优先级和补充建议。
- 围绕订单交易、线索客户、直播、投放、视频内容等高价值主题域，设计 DWD/DWS/ADS 后续建设路线。

**项目成果：**

- 形成全量资产目录、资产盘点摘要、全数仓资产整理文档三类交付物。
- 梳理出核心主题域资产地图，为后续交易营销宽表、线索生命周期宽表、直播经营宽表、投放效果宽表建设提供依据。
- 将原本分散的表资产转化为可治理、可评审、可持续维护的数据资产台账。

### 面试讲述版

这个项目主要解决的是“数仓资产不清晰”的问题。业务侧有很多表，但大家很难知道哪些表属于哪个业务线、哪个主题域、是否有注释、是否能直接用于分析。我先把本地 MaxCompute 表清单和业务线映射文件整理成统一资产目录，然后用 Python 脚本自动识别表的数仓分层、业务线、主题域和注释情况。

最终盘点了 691 张表，发现 ODS 层有 677 张，占比非常高，说明当前更多是源端同步表，DWD/DWS/ADS 建设还不充分。我进一步按业务线和主题域做了分类，识别出 13 个主题域，比如视频内容、线索客户、直播、投放、订单交易、门店 POI 等。基于这些结果，我输出了治理问题清单，比如 19 张未映射表、266 张缺注释表、63 张未分类表，并给出优先治理路径。

这个项目的价值是把“表清单”变成了“数据资产目录”，后续可以继续推进核心主题域建模，比如交易营销宽表、直播经营宽表、投放效果宽表、线索生命周期宽表等。

### 面试可展开亮点

- **资产盘点能力：** 能从表名、注释、业务线映射中自动归类，不只是人工 Excel 维护。
- **数仓分层理解：** 能说明 ODS/DWD/DWS/ADS 当前比例，以及为什么 ODS 过多代表公共层和服务层不足。
- **主题域建模能力：** 能围绕交易、线索、直播、投放、内容设计事实表、维表和汇总宽表。
- **治理意识：** 能识别缺注释、未映射、未分类、临时表、测试表、历史表等治理问题。
- **结果导向：** 交付了资产目录、盘点摘要、全数仓资产整理文档，可用于后续建模和质量治理。

## 项目二：基于 LangChain 的 MaxCompute SQL 智能生成 Agent

### 简历版

**项目名称：基于 LangChain 的 MaxCompute SQL 智能生成 Agent**

**项目背景：**  
数据分析和业务同学经常通过中文问题提出取数需求，但直接编写 MaxCompute SQL 需要熟悉表结构、字段含义、分区规则和 SQL 方言，沟通成本高且容易出现表字段编造、分区遗漏、SQL 不可执行等问题。为提升数据取数效率，设计并实现一个面向 MaxCompute 的 SQL 生成 Agent。

**技术栈：** Python、LangChain、OpenAI API、PyODPS、MaxCompute、dotenv、CLI Agent、Tool Calling

**个人职责：**

- 基于 LangChain `create_agent` 构建命令行 SQL Agent，支持用户输入中文业务问题并生成 MaxCompute SQL。
- 使用 PyODPS 封装 MaxCompute 元数据工具，包括表列表查询、表名关键词搜索、表结构查看、SELECT SQL EXPLAIN 校验。
- 设计 Agent 系统提示词，约束模型必须优先检索真实元数据，禁止凭空编造表结构。
- 增加 SQL 安全边界，默认只允许生成和校验 `SELECT` 查询，拒绝 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`TRUNCATE` 等高风险写操作。
- 针对分区表场景设计查询规范：优先添加分区条件，未指定日期时使用 `${bizdate}` 占位符并提示用户替换。
- 统一 Agent 输出格式，包含使用表、关键字段、SQL、口径假设和风险点，便于人工审阅。

**项目成果：**

- 实现了一个可本地运行的 MaxCompute SQL 生成 CLI Agent。
- 将自然语言需求转化为“查表 → 查字段 → 生成 SQL → EXPLAIN 校验 → 输出口径说明”的半自动化流程。
- 降低了业务取数和数据开发初稿编写成本，减少 SQL 字段编造和分区遗漏风险。

### 面试讲述版

这个项目是我把 Agent 能力和数仓场景结合起来做的一个 SQL 智能生成工具。业务同学通常会说“帮我查某个项目每天的线索数”或者“统计本地生活订单核销情况”，但模型如果直接生成 SQL，最大的问题是容易编造表和字段。所以我没有让大模型直接写 SQL，而是给它接入了 MaxCompute 元数据工具。

我用 LangChain 创建 Agent，用 PyODPS 封装了几个工具：列出表、按关键词搜索表、查看表结构、对 SELECT SQL 做 EXPLAIN 校验。Agent 的系统规则要求它必须先查真实表和字段，再生成 SQL；如果用户没说表名，就先搜索候选表；如果不确定，就输出假设和风险点。

为了安全，我把工具设计成只读模式，默认只处理 SELECT，不执行写操作。输出也不是只给 SQL，而是包含用到的表、关键字段、SQL、口径说明和需要确认的风险点。这个项目体现的是我对 LLM Agent、Tool Calling、数据开发流程和 MaxCompute 生产安全边界的结合能力。

### 面试可展开亮点

- **Agent 工程能力：** 不是简单 ChatBot，而是带工具调用、元数据检索和 SQL 校验的任务型 Agent。
- **数据安全意识：** 默认只允许 SELECT，EXPLAIN 校验也限制 SELECT，避免误执行高风险 SQL。
- **数仓场景结合：** Agent 不是泛泛问答，而是围绕 MaxCompute 表、字段、分区、SQL 方言设计。
- **Prompt 约束能力：** 系统提示词明确要求查元数据、说明假设、暴露风险点。
- **可扩展性：** 后续可接入资产目录、字段血缘、指标口径库、数据质量规则，实现更完整的数据开发 Copilot。

## 两个项目之间的关联讲法

这两个项目可以组合成一条完整主线：

1. 第一个项目解决“数仓资产清不清楚”的问题，把 691 张表整理成业务线、主题域、分层和治理清单。

3. 后续可以把资产目录、主题域文档、指标口径和字段血缘继续接入 Agent，让它从“SQL 生成工具”升级为“数仓研发智能助手”。

面试时可以这样总结：

> 我做的不是单点脚本，而是围绕数据资产治理和 AI Agent 辅助数据开发构建了一套闭环：先盘点和治理数仓资产，再基于真实元数据构建 SQL Agent，让资产能被更安全、更高效地使用。

## AI + 大数据岗位要求提炼

结合公开岗位常见 JD，AI + 大数据方向通常不是只看“会不会调大模型”，而是更看重候选人能否把大模型能力落到数据平台、数仓研发、数据治理和业务分析场景中。核心要求可以归纳为以下几类：

- **大数据基础能力：** 熟悉 SQL、Python、Hive/MaxCompute/Spark/Flink 等数据处理技术，能进行离线/实时数据开发、任务调度、性能优化和数据质量治理。
- **数仓建模能力：** 理解 ODS/DWD/DWS/ADS 分层、维度建模、主题域划分、指标口径管理、宽表建设和数据资产治理。
- **AI 工程落地能力：** 熟悉 LLM、Prompt Engineering、RAG、Agent、Tool Calling、向量检索、知识库问答等能力，并能与企业内部数据系统结合。
- **数据智能应用能力：** 能围绕自然语言取数、指标问答、报表生成、数据资产检索、异常诊断、经营分析等场景设计智能化产品或工具。
- **工程化与安全意识：** 关注权限控制、SQL 安全、元数据约束、日志审计、结果可解释、人工确认机制，避免模型幻觉和高风险操作。
- **业务理解能力：** 能把 AI 能力和交易、投放、直播、线索、内容、用户增长等业务域结合，形成可量化业务价值。

基于这些要求，现有两个项目已经覆盖“数仓资产治理”和“SQL Agent”。为了增强求职竞争力，建议最多新增以下 2 个项目：一个偏 **RAG + 数据资产知识库**，一个偏 **智能指标问答 + 数据分析 Copilot**。

## 项目三：面向数仓资产的 RAG 知识库与智能检索系统

### 简历版

**项目名称：面向数仓资产的 RAG 知识库与智能检索系统**

**项目背景：**  
数仓资产盘点后，表清单、字段说明、业务线归属、主题域分类、宽表设计、ER 关系和数据资产文档分散在多个 Markdown/CSV 文件中。数据开发和分析人员在找表、理解字段、确认口径时仍需要人工翻文档，检索效率低，且容易遗漏上下游关系和治理信息。因此基于已有数仓资产文档构建 RAG 知识库，让用户可以通过自然语言查询数仓资产和业务口径。

**技术栈：** Python、LangChain、RAG、Embedding、向量检索、Markdown/CSV 文档解析、MaxCompute 元数据、数据资产治理

**个人职责：**

- 整理 `warehouse_data_asset_inventory.md`、`table_asset_inventory.csv`、主题域设计文档和 ER 文档，构建统一的数仓资产知识语料。
- 设计文档切分策略，将表资产、字段说明、业务线、主题域、分层、治理问题、宽表设计等内容拆分为可检索知识块。
- 基于 Embedding 和向量检索实现自然语言查表能力，支持“本地生活订单相关表有哪些”“直播主题域有哪些核心表”“哪些表缺少中文说明”等问题。
- 在检索结果中返回表名、分层、业务线、主题域、说明、治理状态和相关文档来源，提升结果可解释性。
- 将 RAG 检索结果与 SQL Agent 结合，让 Agent 在生成 SQL 前先检索资产知识库，再查询 MaxCompute 元数据，降低表字段幻觉风险。

**项目成果：**

- 将静态资产文档升级为可问答、可检索、可复用的数仓知识库。
- 打通“资产盘点 → 知识检索 → SQL 生成”的链路，为数据开发 Copilot 提供基础能力。
- 提升数据开发和分析人员找表、理解口径、定位主题域的效率。

### 面试讲述版

这个项目是在数仓资产治理之后继续做的一层智能化能力。前面我已经把 691 张表整理成了资产目录，但如果只是 Markdown 或 CSV，使用起来还是偏人工。因此我把这些文档做成 RAG 知识库，让用户可以直接用自然语言问：“本地生活订单相关表有哪些？”“直播主题域有哪些核心表？”“哪些表缺少注释？”

实现上，我会先解析资产目录、主题域文档、宽表设计和 ER 文档，然后按表、字段、主题域、治理问题等维度切分成知识块，做 Embedding 和向量索引。查询时先召回相关资产信息，再把表名、业务线、主题域、字段说明和文档来源一起返回。这个系统还可以和 SQL Agent 联动：Agent 写 SQL 前先查知识库确认候选表，再查 MaxCompute 元数据确认真实字段，最后生成 SQL。

这个项目体现的是我不仅能做数仓治理，也能把治理成果转化成 AI 数据产品能力。

### 对应岗位要求

- 对应 **RAG / 知识库 / 企业内部知识问答** 要求。
- 对应 **数据资产治理、元数据管理、指标口径沉淀** 要求。
- 对应 **LLM 与企业数据平台结合** 要求。
- 对应 **降低模型幻觉、增强可解释性** 要求。

## 项目四：经营指标智能问答与数据分析 Copilot

### 简历版

**项目名称：经营指标智能问答与数据分析 Copilot**

**项目背景：**  
业务侧在查看本地生活交易、投放、直播、线索和内容经营数据时，经常需要反复向数据同学确认指标口径、取数逻辑和异常原因。传统 BI 报表只能展示固定指标，难以支持灵活追问、口径解释和异常归因。因此设计一个面向经营分析场景的数据分析 Copilot，支持自然语言指标问答、SQL 生成、结果解释和异常分析。

**技术栈：** Python、LangChain、LLM Agent、MaxCompute SQL、指标口径管理、数仓宽表、数据分析、Prompt Engineering

**个人职责：**

- 基于交易营销宽表、直播经营宽表、投放效果宽表、线索生命周期宽表等主题域设计指标问答场景。
- 梳理核心指标口径，包括 GMV、订单数、核销数、线索数、转化率、投放消耗、ROI、直播曝光、直播成交等。
- 设计 Copilot 工作流：识别用户问题 → 匹配指标口径 → 检索可用表 → 生成 SQL → 校验 SQL → 输出分析结论和口径说明。
- 针对异常分析场景，支持按日期、项目、城市、门店、渠道、达人、直播间等维度拆解波动原因。
- 在输出中明确数据口径、筛选条件、统计周期、SQL 查询和风险提示，保证分析结果可复核。

**项目成果：**

- 将自然语言问题转化为可审阅 SQL 和结构化分析结论，降低业务临时取数成本。
- 建立指标口径、数仓宽表和 AI 问答之间的映射关系，提升经营分析效率。
- 让 AI 能从“生成 SQL”进一步升级为“解释指标、分析波动、辅助决策”的数据分析 Copilot。

### 面试讲述版

这个项目更贴近 AI + 大数据岗位里的“数据智能应用”。我不只是让模型写 SQL，而是围绕经营分析场景设计一个 Copilot。比如业务问：“昨天本地生活 GMV 为什么下降？”系统会先识别这是 GMV 指标，再找到对应的交易宽表和指标口径，生成按日期、项目、城市、渠道等维度拆解的 SQL，然后输出可能的波动原因和需要继续验证的方向。

这里的重点不是让大模型直接给结论，而是把它放在受控流程里：指标口径来自配置或文档，表和字段来自数仓元数据，SQL 只做 SELECT 并经过校验，最终结果要说明统计周期、过滤条件和风险点。这样既能提升分析效率，又能避免模型凭空解释。

这个项目能体现我对数仓、指标体系、业务分析和 LLM Agent 工程化落地的综合能力。

### 对应岗位要求

- 对应 **AI 数据分析、智能 BI、自然语言取数** 要求。
- 对应 **指标体系建设、经营分析、业务理解** 要求。
- 对应 **Agent 工作流、SQL 生成、结果解释** 要求。
- 对应 **数据安全、口径可解释、人工复核** 要求。

面向本地生活交易、投放、直播、线索等经营分析场景，设计自然语言指标问答 Copilot；梳理 GMV、订单数、核销数、线索数、转化率、ROI 等核心指标口径，构建“问题识别 → 指标匹配 → 表检索 → SQL 生成 → SQL 校验 → 结论解释”的 Agent 工作流；支持按日期、城市、项目、渠道等维度拆解异常波动，提升业务取数和经营分析效率。
